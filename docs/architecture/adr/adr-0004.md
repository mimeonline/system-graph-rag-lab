# ADR-0004 Serverless konsistentes Rate Limiting auf Vercel

## Status
1. Accepted am 2026-02-25.

## Kontext
1. Das MVP lﾃ､uft serverless auf Vercel mit mehreren parallelen Instanzen.
2. In Memory Rate Limiting pro Instanz ist nicht konsistent und verletzt die Guardrail von 10 Requests pro 60 Sekunden je Client.
3. Der API Contract verlangt bei Limitﾃｼberschreitung `429 RATE_LIMIT` mit `Retry-After` Header und `error.retryAfterSeconds` im Body.
4. Die Entscheidung muss sofort implementierbar sein und ohne Scope Erweiterung im MVP bleiben.

## Entscheidung
1. Fﾃｼr den MVP wird ein zentraler, persistenter Rate Limit Store ﾃｼber Vercel KV genutzt.
2. Algorithmus ist ein fester Window Counter pro `clientKey` und Route `/api/query`.
3. `clientKey` wird aus der Client IP abgeleitet und gehasht gespeichert, nicht als Klartext geloggt oder persistiert.
4. Das Limit bleibt `10` Requests pro `60` Sekunden je `clientKey`, konfigurierbar ﾃｼber `RATE_LIMIT_MAX_REQUESTS` und `RATE_LIMIT_WINDOW_SECONDS`.
5. Pro Request wird der Counter atomar erhﾃｶht und mit TTL des aktiven Fensters gefﾃｼhrt.
6. Wenn `count > limit`, antwortet die API mit `429 RATE_LIMIT`.
7. `Retry-After` Header und `error.retryAfterSeconds` verwenden denselben ganzzahligen TTL Wert des aktiven Fensters.
8. Bei erfolgreicher Anfrage liefert `meta.rateLimit.remaining` den verbleibenden Wert als `max(0, limit - count)`.

## Begrﾃｼndung
1. Ein zentraler KV Store liefert konsistentes Verhalten ﾃｼber alle serverless Instanzen.
2. Fixed Window ist fﾃｼr das MVP einfach, nachvollziehbar und fﾃｼr QA deterministisch testbar.
3. Atomare Counter Updates verhindern Race Conditions bei gleichzeitigen Requests.
4. Die Lﾃｶsung bleibt minimal und passt zu Vercel als bestehendem Deployment Target.
5. Gehashte Client Schlﾃｼssel reduzieren Datenschutzrisiko gegenﾃｼber Klartext IP Speicherung.

## Konsequenzen
1. Der API Handler benﾃｶtigt eine KV Verbindung und eine atomare Increment plus TTL Operation im Requestpfad.
2. Deployment benﾃｶtigt zusﾃ､tzliche Runtime Variablen fﾃｼr KV Zugriff.
3. QA kann konsistente Grenzfﾃ､lle prﾃｼfen: `10` erfolgreiche Requests, `11.` Request mit `429` und valider Wartezeit.
4. Observability muss bei jedem Request `rateLimitTriggered` korrekt setzen und bei `429` den Retry Wert nachvollziehbar machen.
5. Das Verfahren begrenzt Burst Traffic pro Fenster, ist aber kein Sliding Window.

## Alternativen
1. In Memory Counter im Handler verworfen, da nicht instanzﾃｼbergreifend konsistent.
2. Neo4j basierte Counter verworfen, da zusﾃ､tzliche Last auf dem Graph Backend und unklare Trennung von Domﾃ､nendaten und Betriebszustand.
3. Komplexere Sliding Window oder Token Bucket Varianten verworfen, da fﾃｼr MVP unnﾃｶtig und hﾃｶherer Implementierungsaufwand.
