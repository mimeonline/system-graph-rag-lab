# ADR-0003 Tech Stack und API Implementierungsgrenze

## Status
1. Accepted am 2026-02-25.

## Kontext
1. Für den Dev Start muss eindeutig sein, wo Web UI und API Layer laufen.
2. Das MVP soll ohne zusätzlichen API Service umsetzbar bleiben.
3. QA benötigt eine klare technische Grenze für API Tests und Observability Checks.
4. Die Tech Stack Invarianten aus AGENTS müssen verbindlich und testbar in der Architektur verankert sein.
5. Secret Handling und Programmiersprache müssen für Dev und QA eindeutig prüfbar festgelegt sein.

## Entscheidung
1. Web und API Stack sind verbindlich auf Next.js `16.1.6` als eine Laufzeiteinheit festgelegt.
2. UI Implementierung ist verbindlich mit Tailwind CSS und shadcn/ui festgelegt.
3. UI Architekturpattern ist verbindlich Atomic Design.
4. Der API Layer wird als Next.js Route Handler für `POST /api/query` implementiert.
5. Es gibt keinen separaten API Microservice im MVP.
6. Deploy Target ist Vercel mit Node.js Runtime für den Route Handler.
7. Graph Connector Basis ist der Neo4j Driver gegen Neo4j Aura.
8. LLM Connector Basis ist die OpenAI API über serverseitigen HTTPS Aufruf.
9. Minimale Observability erfolgt über strukturierte JSON Logs im Vercel Runtime Log Stream.
10. Verbindliche Logfelder sind `requestId`, `route`, `method`, `statusCode`, `latencyMs`, `topK`, `hopDepth`, `retrievedNodeCount`, `contextTokens`, `rateLimitTriggered`, `errorCode`.
11. Next.js Anwendung wird verbindlich mit TypeScript umgesetzt, inklusive Route Handlern und Shared Modulen.
12. Secrets und Keys werden ausschließlich aus Environment Variables geladen, lokal über `.env.local` und optional `.env`, public über Vercel Environment Variables.
13. `.env` und `.env.local` dürfen nicht im Repository eingecheckt werden.
14. TypeScript `strict=true` ist als MVP Default verbindlich und darf nicht deaktiviert werden.
15. `OPENAI_MODEL` ist eine verpflichtende Environment Variable; der MVP Defaultwert ist `gpt-5-mini`.
16. Der Modellname darf im Anwendungscode nicht hartkodiert werden und wird ausschließlich aus `process.env.OPENAI_MODEL` gelesen.

## Begründung
1. Eine einzige Next.js Einheit reduziert Integrationsaufwand und beschleunigt MVP Lieferung.
2. Route Handler in Next.js halten den API Contract explizit und testbar ohne zusätzliche Infrastruktur.
3. Die explizite Versionsbindung reduziert Integrationsdrift zwischen Dev, QA und Deploy.
4. Vercel Runtime Logs decken die MVP Observability Anforderungen ohne neues Betriebsmodul.
5. Eine explizite TypeScript Bindung reduziert Integrationsfehler an API und Retrieval Grenzen.
6. Ein festes Env Profil über `.env.local` oder `.env` lokal und Vercel Variablen public verhindert Secret Drift.
7. TypeScript Strictness reduziert Laufzeitrisiken an den API und Retrieval Vertragsgrenzen.
8. Ein Environment basierter Model Default erlaubt kontrollierte Modellwechsel ohne Codeänderung.

## Konsequenzen
1. API und UI Versionierung bleiben immer synchron im selben Deploy.
2. Rate Limiting und Fehlerbehandlung müssen serverless kompatibel im Route Handler umgesetzt werden.
3. Observability bleibt auf Request Ebene und enthält keine erweiterten Traces im MVP.
4. Rohquery Inhalte und Secrets dürfen nicht in Logs erscheinen.
5. Abweichungen von Next.js `16.1.6`, Tailwind CSS, shadcn/ui oder Atomic Design erfordern eine neue ADR Entscheidung.
6. Dev Implementierung muss für neue Module standardmäßig `.ts` oder `.tsx` nutzen.
7. Lokale Startfähigkeit setzt gültige Env Dateien voraus, ohne Secret Werte im Repository.
8. TypeScript Build Checks müssen mit `strict=true` erfolgreich laufen.
9. Fehlendes `OPENAI_MODEL` muss früh fehlschlagen, statt implizit auf einen hartkodierten Modellnamen zurückzufallen.

## Verworfene Alternativen
1. Separater API Service auf eigener Runtime erhöht Betriebsaufwand und widerspricht MVP Vereinfachung.
2. Externes Logging System im MVP erhöht Setup Komplexität ohne zwingenden Mehrwert.
