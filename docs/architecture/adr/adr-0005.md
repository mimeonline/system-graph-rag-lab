# ADR-0005 Local Development Topologie mit Next.js lokal und Neo4j Docker

## Status
1. Accepted am 2026-02-25.

## Kontext
1. Public Runtime bleibt verbindlich auf Vercel plus Neo4j Aura mit Vercel KV.
2. Dev benötigt einen reproduzierbaren lokalen Startpfad ohne Abhängigkeit von Vercel Runtime oder Neo4j Aura.
3. API und Retrieval Contracts müssen zwischen Local Dev und Public Runtime identisch bleiben.
4. Unterschiede zwischen Local Dev und Public Runtime müssen explizit dokumentiert und testbar getrennt sein.

## Entscheidung
1. Local Dev nutzt dieselbe Next.js `16.1.6` Anwendung lokal auf `http://localhost:3000` mit identischer API Grenze `POST /api/query`.
2. Local Dev nutzt Neo4j in einem lokalen Docker Container als Graph und Vektor Backend.
3. Public Runtime bleibt unverändert mit Neo4j Aura als produktivem Graph Backend.
4. Local Dev nutzt für Rate Limiting einen prozesslokalen Fixed Window Store mit denselben Contractwerten wie Public Runtime.
5. Public Runtime nutzt weiterhin Vercel KV als zentralen Fixed Window Store gemäß ADR-0004.
6. Contract Parameter bleiben in beiden Profilen identisch: `TopK=6`, `HopDepth=1`, `ContextBudget=1400`, Fehlercodes und Observability Pflichtfelder.
7. Local Dev gilt als deterministisch reproduzierbar, wenn Ports, Environment-Profil und lokale Datenbasis stabil und dokumentiert sind.
8. Local Neo4j Docker wird im MVP auf das feste Image Tag `neo4j:5.26.0` gepinnt.
9. Das Tag `latest` ist für Local Dev unzulässig.
10. Ein Wechsel des Neo4j Local Tags erfordert ADR Update und erneuten Determinismusnachweis in QA.

## Begründung
1. Lokales Next.js plus Docker Neo4j ermöglicht Dev Start und Kernentwicklung ohne Abhängigkeit von Public Infrastruktur.
2. Gleiches API Surface und gleicher Retrieval Contract reduzieren Integrationsdrift zwischen Local Dev, QA und Public Runtime.
3. Ein lokaler Rate Limit Store vermeidet lokale Abhängigkeit von Vercel KV, ohne den Public Contract zu verändern.
4. Die klare Profiltrennung verhindert Scope Drift und hält das Produktionsziel unverändert.
5. Ein fixes Image Tag minimiert lokale Drift gegenüber wiederholbaren QA Replays.

## Konsequenzen
1. Architektur- und Spezifikationsdokumente müssen Laufzeitprofile explizit trennen: `public` und `local`.
2. Dev muss lokale Neo4j Laufzeit inklusive Datenmodell und Vektorindex konsistent zu Aura halten.
3. QA muss Contract Tests in beiden Profilen mit identischen Akzeptanzkriterien ausführen.
4. Unterschiede bei Rate Limit Persistenz sind lokal erlaubt, solange Request und Response Contract identisch bleiben.
5. OpenAI API bleibt externe Abhängigkeit für echte Antwortgenerierung, ist aber kein blocker für lokalen Infrastrukturstart.
6. Dev Setup und CI lokale Reproduktion müssen dasselbe Neo4j Image Tag nutzen.

## Verworfene Alternativen
1. Local Dev direkt gegen Neo4j Aura wurde verworfen, weil es den lokalen Start von externer Infrastruktur abhängig macht.
2. Local Dev mit separatem API Service wurde verworfen, weil es der MVP Architektur als eine Next.js Einheit widerspricht.
3. Local Dev mit deaktiviertem Rate Limit wurde verworfen, weil es das Fehlerverhalten gegenüber Public Runtime verfälscht.
